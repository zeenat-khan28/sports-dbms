1. INTRODUCTION TO DATA FLOW DIAGRAMS
--------------------------------------
Data Flow Diagrams (DFDs) provide a visual representation of how data moves
through the system, showing processes, data stores, external entities, and
the flow of information between them.

NOTATION KEY:
=============
┌─────────┐
│ Process │  = Rectangle: Process/Function that transforms data
└─────────┘

┌═════════┐
║External ║  = Double rectangle: External entity (user, system)
║ Entity  ║
└═════════┘

│─────────│
│Data Store│ = Open rectangle: Database/File storage
│─────────│

   ───▶     = Arrow: Data flow direction

LEVELS OF DFD:
--------------
Level 0 (Context Diagram): Highest level overview - system as single process
Level 1: Major subsystems and their interactions
Level 2: Detailed breakdown of individual subsystems

EXTERNAL ENTITIES:
------------------
1. STUDENT: External user interacting via web browser (Student Portal)
2. ADMIN: System administrators managing the platform (Admin Portal)
3. EMAIL SYSTEM: SMTP server for sending notifications (External service)

MAJOR DATA FLOWS:
-----------------
FROM STUDENT:
  - Login Credentials (Google OAuth)
  - Registration Form (Personal details, documents)
  - Event Participation Requests

TO STUDENT:
  - Approval Status (Approved/Rejected/Pending)
  - Event List (Available competitions)
  - Selection Status (Selected/Dropped for events)
  - Email Notifications

FROM ADMIN:
  - Login Credentials (Email/Password)
  - Event Creation Data
  - Approval/Rejection Actions
  - Attendance Marking
  - Bulk Email Content

TO ADMIN:
  - Dashboard Statistics
  - Analytics Reports
  - Pending Approvals List
  - Student Submissions

TO EMAIL SYSTEM:
  - Bulk email content and recipient lists

FROM EMAIL SYSTEM:
  - Delivery status and receipts

LEVEL 1 PROCESSES:
------------------

1.0 STUDENT REGISTRATION MANAGEMENT
   - Input: Student registration forms (from Student)
   - Output: Approval/rejection status (to Student)
   - Data Stores: student_submissions (MongoDB), students (PostgreSQL)

2.0 AUTHENTICATION & AUTHORIZATION
   - Input: Login credentials (from Student/Admin)
   - Output: JWT tokens, session data
   - Data Stores: users (PostgreSQL)

3.0 EVENT MANAGEMENT
   - Input: Event details (from Admin)
   - Output: Event list (to Student/Admin)
   - Data Stores: events (PostgreSQL)

4.0 PARTICIPATION MANAGEMENT
   - Input: Participation requests (from Student), Selection actions (from Admin)
   - Output: Selection status (to Student)
   - Data Stores: event_participation_requests (MongoDB), approved_participants (PostgreSQL)

5.0 ATTENDANCE TRACKING
   - Input: Attendance records (from Admin)
   - Output: Attendance reports (to Admin)
   - Data Stores: event_attendance (PostgreSQL)

6.0 EMAIL COMMUNICATION
   - Input: Email content and filters (from Admin)
   - Output: Email delivery (via SMTP), Audit logs
   - Data Stores: email_audit_logs (PostgreSQL)

7.0 ANALYTICS & REPORTING
   - Input: Analytics requests (from Admin)
   - Output: Dashboard metrics, charts, trends
   - Data Stores: All database tables (read-only queries)

SUB-PROCESSES:

1.1 AUTHENTICATE STUDENT
   - Input: Google OAuth token (from Student)
   - Process: Verify token with Firebase, check @rvce.edu.in domain
   - Output: Verified student identity

1.2 SUBMIT REGISTRATION FORM
   - Input: Student details, Base64 documents
   - Process: Validate format, convert files, create MongoDB document
   - Output: Submission ID, confirmation message

1.3 RETRIEVE PENDING SUBMISSIONS
   - Input: Admin request for pending list
   - Process: Query MongoDB where status='pending'
   - Output: List of pending submissions with metadata

1.4 VALIDATE & REVIEW SUBMISSION
   - Input: Submission details
   - Process: Admin manually reviews data, documents
   - Output: Approval/rejection decision

1.5 PROCESS APPROVAL
   - Input: Approval decision
   - Process: Update MongoDB status to 'approved' or 'rejected'
   - Output: Updated submission record

1.6 MIGRATE TO POSTGRESQL
   - Input: Approved submission data
   - Process: Extract fields, insert into students table
   - Output: New student record in PostgreSQL

1.7 SEND EMAIL NOTIFICATION
   - Input: Approval result, student email
   - Process: Generate email content, call SMTP service
   - Output: Email sent to student

SUB-PROCESSES:

4.1 DISPLAY AVAILABLE EVENTS
   - Input: Student request
   - Process: Query events table for upcoming events
   - Output: Event list with details

4.2 VALIDATE ELIGIBILITY
   - Input: Student ID, Event ID
   - Process: Check if student is approved in student_submissions
   - Output: Eligibility status

4.3 CREATE PARTICIPATION REQUEST
   - Input: Student USN, Event ID
   - Process: Create document in event_participation_requests
   - Output: Request ID, pending status

4.4 RETRIEVE REQUESTS FOR EVENT
   - Input: Event ID, optional status filter
   - Process: Query MongoDB for requests matching event
   - Output: List of participation requests

4.5 SELECT/DROP PARTICIPANTS
   - Input: Request IDs, decision (selected/dropped)
   - Process: Admin reviews and decides
   - Output: Selection decisions

4.6 GENERATE BLOCKCHAIN HASH
   - Input: USN, Event ID, Admin ID, Timestamp
   - Process: Compute SHA-256 hash
   - Output: Cryptographic hash for audit

4.7 UPDATE REQUEST STATUS
   - Input: Request ID, new status, hash
   - Process: Update MongoDB document
   - Output: Updated request with blockchain hash

4.8 CREATE APPROVED PARTICIPANT RECORD
   - Input: Selected participant data
   - Process: Insert record into PostgreSQL approved_participants
   - Output: New approved participant record

4.9 NOTIFY STUDENT
   - Input: Selection result, student email
   - Process: Generate email, send via SMTP
   - Output: Email notification to student

7. DATA FLOW SUMMARY
================================================================================

CRITICAL DATA PATHS:
--------------------

1. STUDENT ONBOARDING FLOW:
   Student → Authentication → Registration Form → MongoDB →
   Admin Review → Approval → PostgreSQL → Email Notification

2. EVENT PARTICIPATION FLOW:
   Admin Creates Event → PostgreSQL → Student Views Events →
   Request to Join → MongoDB → Admin Selects → PostgreSQL →
   Blockchain Hash → Email Notification

3. ATTENDANCE FLOW:
   Admin Opens Attendance → Load Participants → Mark Attendance →
   PostgreSQL → Analytics Update

4. EMAIL COMMUNICATION FLOW:
   Admin Composes Email → Apply Filters → Query Students →
   Send via SMTP → Log Audit Trail → PostgreSQL

5. ANALYTICS FLOW:
   Admin Request → Query All Databases → Aggregate Data →
   Calculate Metrics → Format Charts → Render Dashboard


DATA STORE ACCESS PATTERNS:
----------------------------

WRITE-HEAVY:
- student_submissions (MongoDB): Continuous student registrations
- event_participation_requests (MongoDB): High-volume participation requests
- email_audit_logs (PostgreSQL): Every email logged

READ-HEAVY:
- events (PostgreSQL): Frequent queries by students
- students (PostgreSQL): Name lookups, email filtering
- Analytics queries: Cross-table aggregations

HYBRID (Read/Write):
- approved_participants (PostgreSQL): Frequent inserts, moderate reads
- event_attendance (PostgreSQL): Daily writes, frequent reads for reports
