1. INTRODUCTION
================================================================================

1.1 PURPOSE
-----------
This Software Requirements Specification (SRS) document provides a complete
description of the RVCE Sports Management System. It details functional and
non-functional requirements, system interfaces, and design constraints.

The system digitizes and automates the entire sports activity management
lifecycle at RV College of Engineering, replacing manual paperwork with a
centralized, secure, and scalable web application.

1.2 SCOPE
---------
Project Name: RVCE Sports Management System

The system encompasses:
- Student registration and profile management
- Sports event creation and scheduling
- Participant selection and enrollment
- Attendance tracking and monitoring
- Email communication and notifications
- Analytics and reporting dashboards
- Admin authentication and authorization
- Audit trail with blockchain simulation

Benefits:
- Eliminates manual paperwork and data entry
- Provides real-time insights into sports participation
- Ensures data integrity and auditability
- Enables efficient communication with students
- Streamlines event management workflows

--------------------
The RVCE Sports Management System is a full-stack web application built with:

3. FUNCTIONAL REQUIREMENTS
================================================================================

3.1 STUDENT REGISTRATION MANAGEMENT
------------------------------------

FR-101: Student Authentication via Google OAuth
Priority: HIGH | Type: Functional

Description:
The system shall allow students to authenticate using their Google accounts
through Firebase OAuth integration.

Inputs:
- Google OAuth token from Firebase SDK
- Student email from Google account

Processing:
1. Validate OAuth token with Firebase
2. Extract email from token claims
3. Verify email ends with "@rvce.edu.in"
4. Create or retrieve student session

Outputs:
- Authenticated student session
- Error message if domain invalid

Acceptance Criteria:
- Only @rvce.edu.in emails accepted
- Invalid domain shows clear error message
- Session persists across page refreshes


FR-102: Student Registration Form Submission
Priority: HIGH | Type: Functional

Description:
The system shall provide a comprehensive registration form for students to
submit their details and documents.

Inputs:
- Personal Information: Name, USN, Branch, Semester, DOB, Blood Group
- Contact Information: Phone, Address, Parent names
- Academic Information: Course, admission dates, previous participation
- Documents: Photo (JPEG/PNG), Signature (JPEG/PNG)
- Sports Information: Game/Sport/Competition, organizing institution, dates

Processing:
1. Validate all required fields
2. Convert uploaded files to Base64 encoding
3. Validate USN format (pattern: 1RVYYBBXXX)
4. Check for duplicate USN in MongoDB
5. Create document in student_submissions collection
6. Set status to "pending"

Outputs:
- Submission confirmation message
- Unique submission ID
- "Pending Approval" status displayed

Acceptance Criteria:
- All mandatory fields must be filled
- Photo and signature files must be < 5MB each
- USN must be unique (duplicate prevention)
- Form data persisted even if browser closes


FR-103: Admin Review and Approval
Priority: HIGH | Type: Functional

Description:
The system shall allow admins to review pending registrations and approve
or reject them with reasons.

Inputs:
- Admin selection of pending submission
- Approval decision (Approve/Reject)
- Rejection reason (if rejecting)

Processing:
1. Display submission details with photo and signature
2. If approved:
   a. Extract data from MongoDB document
   b. Insert record into PostgreSQL students table
   c. Update MongoDB status to "approved"
   d. Send approval email to student
3. If rejected:
   a. Update MongoDB status to "rejected"
   b. Store rejection reason
   c. Send rejection email with reason

Outputs:
- Updated submission status
- Student record in PostgreSQL (if approved)
- Email notification sent to student

Acceptance Criteria:
- Admin can view all pending submissions
- Rejection requires reason (minimum 10 characters)
- Approval migrates data to PostgreSQL
- Email notification sent within 5 seconds


FR-104: View Registration Status
Priority: MEDIUM | Type: Functional

Description:
Students shall be able to view their registration status in real-time.

Inputs:
- Student authentication token

Processing:
1. Query MongoDB student_submissions by student email/UID
2. Retrieve current status and metadata
3. Display status badge (Pending/Approved/Rejected)
4. Show rejection reason if applicable

Outputs:
- Status badge with color coding:
  * Pending: Yellow/Orange
  * Approved: Green
  * Rejected: Red
- Rejection reason (if rejected)
- Submission timestamp

Acceptance Criteria:
- Status updates reflect immediately after admin action
- Rejection reason clearly visible
- Students can resubmit after rejection (with new USN if needed)


3.2 EVENT MANAGEMENT
--------------------

FR-201: Create Sports Event
Priority: HIGH | Type: Functional

Description:
Admins shall be able to create new sports events with details and date ranges.

Inputs:
- Event name (required, min 3 characters)
- Location (required)
- Start date (required, date picker)
- End date (required, must be >= start date)
- Description (optional, rich text)

Processing:
1. Validate date range (end >= start)
2. Validate name uniqueness (optional constraint)
3. Insert record into PostgreSQL events table
4. Link created_by to admin user ID
5. Set created_at timestamp

Outputs:
- New event record in database
- Event ID generated
- Success confirmation message

Acceptance Criteria:
- End date cannot be before start date
- Event name minimum 3 characters
- Future or current dates only (no past events)
- Created event immediately visible to students


FR-202: View Available Events (Student)
Priority: HIGH | Type: Functional

Description:
Students shall be able to view all available sports events.

Inputs:
- Student authentication token
- Optional filters: upcoming, ongoing, past

Processing:
1. Query PostgreSQL events table
2. Order by start_date DESC
3. Calculate event status (upcoming/ongoing/ended)
4. Display event cards with details

Outputs:
- List of events with:
  * Name, location, date range
  * Description
  * Status badge
  * Participant count (if applicable)
  * "Request to Join" button (if eligible)

Acceptance Criteria:
- Events sorted by start date (newest first)
- Past events clearly marked as "Ended"
- Students cannot request participation in ended events


FR-203: Edit/Delete Event
Priority: MEDIUM | Type: Functional

Description:
Admins shall be able to edit event details or delete events.

Inputs:
- Event ID
- Updated event fields (name, location, dates, description)
- OR delete confirmation

Processing:
1. For Edit:
   a. Validate updated fields
   b. Update PostgreSQL event record
   c. Show success message
2. For Delete:
   a. Check for existing participants (cascade setting)
   b. Delete event (cascades to approved_participants, attendance)
   c. Confirm deletion

Outputs:
- Updated event record (if edited)
- Deletion confirmation (if deleted)
- Associated records cascade-deleted

Acceptance Criteria:
- Date constraints still enforced on edit
- Delete shows warning if participants exist
- Cascade deletion removes all related records


3.3 PARTICIPATION MANAGEMENT
-----------------------------

FR-301: Request Event Participation
Priority: HIGH | Type: Functional

Description:
Approved students shall be able to request participation in events.

Inputs:
- Student USN (from authenticated session)
- Event ID
- Student name (from registration)

Processing:
1. Verify student is approved in student_submissions
2. Check event hasn't ended
3. Check for duplicate request (usn + event_id)
4. Create document in event_participation_requests (MongoDB)
5. Set status to "pending"

Outputs:
- Participation request created
- "Pending" status displayed
- Request ID generated

Acceptance Criteria:
- Only approved students can request
- Cannot request for same event twice
- Cannot request after event ended
- Request immediately visible in student's activity


FR-302: Admin Select/Drop Participants
Priority: HIGH | Type: Functional

Description:
Admins shall be able to review requests and select or drop participants.

Inputs:
- Event ID
- List of request IDs
- Action (Select/Drop)

Processing:
1. Retrieve participation requests for event
2. For each selected request:
   a. Generate blockchain hash (SHA-256)
   b. Update MongoDB status to "selected"
   c. Store blockchain_hash and processed_by
   d. Create record in PostgreSQL approved_participants
   e. Send email notification to student
3. For dropped requests:
   a. Update status to "dropped"
   b. No email sent (silent drop)

Outputs:
- Updated request statuses in MongoDB
- New approved_participant records in PostgreSQL
- Email notifications sent to selected students
- Blockchain hashes generated for audit

Acceptance Criteria:
- Admin can bulk-select multiple students
- Blockchain hash unique for each selection
- Email sent within 5 seconds of selection
- Selected students visible in event roster


FR-303: View Participation Status (Student)
Priority: MEDIUM | Type: Functional

Description:
Students shall view status of all their participation requests.

Inputs:
- Student USN

Processing:
1. Query event_participation_requests by USN
2. Join with events table for event details
3. Order by submitted_at DESC

Outputs:
- Table of participation requests showing:
  * Event name
  * Submission date
  * Status (Pending/Selected/Dropped)
  * Blockchain hash (if selected)

Acceptance Criteria:
- All requests visible regardless of status
- Status updates reflect in real-time
- Selected events highlighted with green badge


3.4 ATTENDANCE TRACKING
-----------------------

FR-401: Mark Event Attendance
Priority: HIGH | Type: Functional

Description:
Admins shall be able to mark daily attendance for event participants.

Inputs:
- Event ID
- Attendance date
- USN and attendance status (Present/Absent) for each participant

Processing:
1. Retrieve approved participants for event
2. Validate attendance_date within event date range
3. For each participant:
   a. Check for existing attendance record (duplicate prevention)
   b. Insert or update event_attendance record
   c. Set marked_by to admin user ID
   d. Timestamp with marked_at

Outputs:
- Attendance records created/updated
- Success message with count (e.g., "Attendance marked for 25 students")

Acceptance Criteria:
- Only approved participants listed
- Attendance date must be within event range
- Duplicate attendance prevented (unique constraint)
- Bulk marking saves all records atomically


FR-402: View Attendance Report
Priority: MEDIUM | Type: Functional

Description:
Admins shall be able to view and export attendance reports for events.

Inputs:
- Event ID
- Optional date filter

Processing:
1. Query event_attendance table for event
2. Group by attendance_date
3. Calculate present/absent counts
4. Support export to Excel

Outputs:
- Attendance table showing:
  * Date
  * Total participants
  * Present count
  * Absent count
  * Attendance percentage
- Export button for Excel download

Acceptance Criteria:
- Attendance percentage calculated correctly
- Export includes all columns
- Report updates immediately after marking


3.5 EMAIL COMMUNICATION
-----------------------

FR-501: Send Bulk Emails with Filtering
Priority: HIGH | Type: Functional

Description:
Admins shall be able to send bulk emails to students with filter criteria.

Inputs:
- Email subject (required, max 255 chars)
- Email body (required, supports HTML)
- Filters: Branch (multiselect), Semester (multiselect), USNs (optional)

Processing:
1. Validate subject and body not empty
2. Apply filters to query students table
3. Retrieve matching student emails
4. For each recipient:
   a. Replace placeholders ({{name}}, {{usn}})
   b. Send via SMTP
   c. Track success/failure
5. Create record in email_audit_logs
6. Store filters as JSON, log success/failure counts
7. Generate blockchain hash for audit

Outputs:
- Email sent to filtered recipients
- Audit log entry with recipient count and filters
- Success message: "Email sent to N students"

Acceptance Criteria:
- Minimum 1 recipient required
- Placeholder replacement works correctly
- Audit log captures all send attempts
- Failed emails logged with reason


FR-502: View Email Audit Logs
Priority: MEDIUM | Type: Functional

Description:
Admins shall be able to view history of all bulk emails sent.

Inputs:
- Admin authentication
- Optional date filter

Processing:
1. Query email_audit_logs table
2. Order by sent_at DESC
3. Display paginated results (50 per page)

Outputs:
- Table showing:
  * Sent date/time
  * Subject
  * Filters used
  * Recipient count
  * Success/Failure counts
  * Sent by (admin name)

Acceptance Criteria:
- Logs retained indefinitely (or per policy)
- Filters displayed as human-readable text
- Audit records immutable (no deletion allowed)

3.7 SECURITY AND AUTHENTICATION
--------------------------------

FR-701: Admin Login with JWT
Priority: HIGH | Type: Security

Description:
System shall authenticate admins using email/password with JWT tokens.

Inputs:
- Admin email
- Admin password (plaintext from form)

Processing:
1. Query users table by email
2. Compare password with bcrypt hash
3. If valid:
   a. Generate JWT token (expires in 24 hours)
   b. Include user ID and email in claims
   c. Return token to frontend
4. If invalid:
   a. Return 401 Unauthorized error

Outputs:
- JWT access token (on success)
- Error message (on failure)

Acceptance Criteria:
- Passwords hashed with bcrypt (cost factor 12)
- JWT includes expiration claim
- Token stored in localStorage or cookie (frontend)
- Token validated on all protected endpoints


FR-702: Role-Based Access Control (RBAC)
Priority: HIGH | Type: Security

Description:
System shall enforce access control based on user roles.

Roles:
- STUDENT: Can access student portal endpoints only
- ADMIN: Can access all admin endpoints
- SUPERUSER: Can manage admin users (future)

Processing:
1. Extract role from JWT token (for admin) or Firebase token (for student)
2. Check required role for endpoint
3. Allow or deny access

Outputs:
- 200 OK if authorized
- 403 Forbidden if unauthorized

Acceptance Criteria:
- Students cannot access admin endpoints (403 response)
- Admins cannot access student registration
- Role checked on every protected endpoint


FR-703: Blockchain Audit Trail
Priority: MEDIUM | Type: Security

Description:
System shall generate cryptographic hashes for critical actions (approvals,
 selections) to ensure audit integrity.

Inputs:
- Action data (USN, event ID, admin ID, timestamp, action type)

Processing:
1. Concatenate input fields into string
2. Compute SHA-256 hash
3. Store hash in database (blockchain_hash fields)
4. Link hash to previous hash (future blockchain chain)

Outputs:
- 64-character hexadecimal hash
- Hash stored in approved_participants or email_audit_logs

Acceptance Criteria:
- Hash generated for every approval and selection
- Hash immutable once created
- Hash can be verified by recomputing from stored data
